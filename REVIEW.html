1. Overview & Changed Files

When a PR fails to merge with UnmergablePRError (e.g. branch protection requires more reviews), Tide now temporarily excludes that PR from the merge queue for 5 minutes. This allows other mergeable PRs to proceed instead of blocking the entire queue. The exclusion key includes the PR's head SHA, so pushing new commits automatically clears the exclusion.

3. Code Quality

Senior Go Engineer Perspective

Approve

Critical Issues

None.

Improvements (non-blocking)

  • No periodic cleanup of expired entries 3/3 reviewers — If a PR is excluded then removed from the pool (closed, labels changed), its map entry lingers until restart. Entries are tiny and the expected cardinality is low, so this is theoretical. A periodic sweep at the start of Sync() would close the gap.
  • sync.Mutex vs sync.RWMutex — isExcludedFromMerge is called per-PR per-sync and the common case (not found) is read-only. Technically correct since it also deletes expired entries, but RWMutex with escalation would be more idiomatic. Very minor given low contention.
  • Batch merges not covered by exclusion logic 3/3 reviewers — The batchMerges path at the top of takeAction does not filter out excluded PRs or record exclusions on failure. If a PR fails individual merge due to branch protection, it could still enter a batch merge and fail there.
  • isUnmergableError semantics with batches — Returns true if any error in a mergeFailure is unmergeable. Correct for the single-PR merge path (only caller), but could be misleading if reused for batch merges since it would flag the whole batch without identifying which PR caused it.

Positive Observations

  • SHA-keyed exclusion design is elegant — auto-invalidates on new commits without explicit cleanup logic
  • Proactive prefix-based cleanup in excludeFromMerge prevents unbounded map growth from repeated pushes
  • isUnmergableError correctly handles the error wrapping chain: direct type assertion for *mergeFailure (which doesn't implement Unwrap()), then errors.As for UnmergablePRError (which may be wrapped by fmt.Errorf)
  • Thorough test coverage: integration two-cycle test, table-driven error detection with positive and negative cases, TTL expiry with cleanup verification
  • Excellent comments and documentation throughout, with issue references

4. Maintainability

Long-term Maintainer Perspective

Approve Burden: Low

Concerns

  • Batch merge asymmetry 3/3 reviewers — Exclusions only apply to the single-PR Merge path, not MergeBatch. A comment explaining whether this is intentional or a known gap would help future maintainers. If intentional, explain why; if a gap, file a follow-up.
  • No periodic sweep 3/3 reviewers — Expired entries only cleaned on read (isExcludedFromMerge) or re-exclusion (excludeFromMerge). Negligible with 5-min TTL but could matter if TTL is increased.
  • time.Now() direct usage — Makes time-dependent behavior hard to test without manipulating internal map state directly (which the tests do). Pre-existing pattern in the codebase, so not a regression.
  • Hard-coded TTL — 5 minutes is fine for most deployments, but a comment explaining the design choice (and that it could be made configurable in the future) would help.

Positive Patterns

  • Self-contained: isolated to a single controller struct, no cross-package coupling
  • SHA-based invalidation avoids explicit cleanup logic for the common case (author pushes new commits)
  • TTL is a discoverable package-level constant, easy to find and tune
  • Clear comments explaining "why" throughout, with issue references
  • Cleanly revertible — removing the map, methods, and filtering restores original behavior
  • Tests are behavior-oriented rather than implementation-coupled

5. Deployment Risk

Platform Operator Perspective

Risk: Low

Breaking Changes

None. No config fields, CLI flags, API endpoints, or serialization formats changed.

Behavioral Changes

  • Unmergeable PRs skipped for 5 minutes instead of retried every sync cycle (the intended fix). Strictly an improvement — the PR can't merge anyway during the exclusion window.
  • New Warning-level log when a PR is excluded, Debug-level when skipping. Low volume (at most once per unmergeable PR per 5-min window). Operators with log-based alerting on Tide warnings should be aware.
  • Batch merges are not covered — exclusion only applies to the single-PR merge path.

Upgrade/Rollback

  • No config migration required
  • Safe to roll back at any time (reverts to retry-every-cycle behavior)
  • Exclusion state is in-memory only, cleared on pod restart (desirable)
  • No HA concerns: per-instance tracking, worst case is a harmless duplicate retry

Risk Mitigations

  • SHA-based invalidation auto-clears on new pushes
  • TTL-based expiry — worst case is 5-min delay
  • Narrow error matching — only UnmergablePRError, not network errors or other failures
  • Prefix-based cleanup prevents unbounded map growth from SHA changes
