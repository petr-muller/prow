Verdict

Approve

The hack/tools bump is correct and should merge. It patches a real High-severity DoS (CVE-2026-34986, CVSS 7.5) via a panic in JWE decryption. The only gap worth noting is that the root go.mod still carries the vulnerable v4.1.3 — whether that matters depends on whether prow's runtime touches the affected code path.

What this PR does

  • Bumps go-jose/go-jose/v4 from v4.1.3 → v4.1.4 in hack/tools/go.mod.
  • Updates the corresponding go.sum hashes (both .mod and source entries).
  • Fixes CVE-2026-34986 (GHSA-78h2-9frx-2jm8): a panic triggered in Decrypt() when a JWE specifies a KW-type key-wrapping algorithm and has an empty encrypted_key field — remotely exploitable, no authentication required.
  • v4.1.4 was published 2026-04-04, approximately 6 weeks before this review.
  • No source code changes; diff is dependency metadata only.

Findings

Question Root module still on vulnerable v4.1.3
go.mod (root)

The root go.mod still has:

github.com/go-jose/go-jose/v4 v4.1.3 // indirect

This PR only fixes hack/tools. If prow's runtime exercises the JWE decrypt path — ParseEncrypted() followed by Decrypt() — on user-controlled input, the main binary remains vulnerable. Given prow handles GitHub webhooks and auth flows, it's worth confirming. Even if the path is unexposed, shipping the known-vulnerable version when the fix exists seems unnecessary.

Things I checked and was fine with

  • hack/tools/go.mod: version pin correctly updated 4.1.3 → 4.1.4.
  • hack/tools/go.sum: both the .mod and source hashes replaced with the correct v4.1.4 entries.
  • Diff is dependency metadata only — no logic changes, no risk of behavioral regression.
  • Security advisory confirms this is a real panic (DoS), not theoretical — the panic path is reachable without auth.

Open questions for the author

  • Does prow's runtime use go-jose's JWE decrypt path (ParseEncrypted + Decrypt) on user-controlled input? If yes, a follow-up bump of the root go.mod should be tracked.
