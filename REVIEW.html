What Works Well

Good Fail-closed design via onConfigLoadError callback All 3 reviewers

The callback pattern correctly propagates config-agent goroutine errors back to the controller's health state. This is the right solution for the goroutine boundary problem identified in review feedback. The fail-closed design is sound: corrupted state → Load() fails → Run() returns → liveness check fails → pod restart.

Good Functional options for ConfigAgent Code Quality + Maintainability

WithErrorHandler, WithAdditionals, WithReuseAgent is idiomatic Go. Consolidates ConfigAgent + ConfigAgentWithAdditionals into one method. Extensible for future needs without API breakage. Deck's call site is cleaner.

Good Corrupted state file now fails Load() All 3 reviewers

Only io.IsNotExist errors are tolerated. Previously, the reconciler could silently proceed with empty state on corruption, treating all presubmits as "new" and triggering spurious job runs across every PR. Test coverage added for this path.

Good Metrics are well-scoped and directly address #540 Maintainability

status_reconciler_loaded_presubmit_count (Gauge) gives immediate signal for the "all presubmits disappeared" failure mode. status_reconciler_contexts_retired_total (Counter with org/repo labels) enables alerting on unexpected retirements.

Good StartWatch fix propagates watchConfigs errors Code Quality

Previously StartWatch silently discarded watchConfigs return value. Now it's propagated. Good correctness improvement — if fsnotify setup fails, callers will know.

Good sync/atomic for healthy flag Code Quality + Maintainability

Appropriate for a simple boolean, avoids mutex overhead. Matches the existing ReadinessCheck pattern in the same file.

Converging Concerns

Issues independently flagged by 2+ reviewers. These carry the most weight.

All 3 Liveness probe initialDelaySeconds: 3 is too aggressive config/prow/cluster/starter/starter-*.yaml

The liveness probe has initialDelaySeconds: 3 with periodSeconds: 3. For status-reconciler, ServeLive(c.Healthy) is called after config agent, plugin agent, GitHub client, and controller init. During this window, /healthz returns 404. The kubelet could start probing before the handler exists and restart the pod.

The readiness probe uses initialDelaySeconds: 10, acknowledging startup time. The liveness probe should be at least as generous.

livenessProbe:
  httpGet:
    path: /healthz
    port: 8081
  initialDelaySeconds: 3   # <-- too low if ServeLive() isn't called yet
  periodSeconds: 3

Bump liveness initialDelaySeconds to at least 10 (matching readiness). Consider adding failureThreshold: 3 for additional safety.

CQ + DR ServeLive() registration window — /healthz will 404 until called pkg/pjutil/health.go:46

Previously, NewHealthOnPort registered /healthz unconditionally. Now the handler is only registered when ServeLive() is called. The HTTP server starts in NewHealthOnPort, so there's a window where /healthz returns 404.

For most binaries this window is tiny (ServeLive is called right after). But in cmd/status-reconciler/main.go, ServeLive(c.Healthy) is called much later — a potentially multi-second window.

// main.go order:
health := pjutil.NewHealthOnPort(...)   // server starts, /healthz = 404
health.ServeReady()                     // /healthz/ready registered
configAgent, err := o.config.ConfigAgent()  // may take time
// ... plugin agent, github client, opener ...
c := statusreconciler.NewController(...)
health.ServeLive(c.Healthy)             // /healthz finally registered

The initialDelaySeconds bump above is the primary mitigation. Optionally, update NewHealthOnPort godoc to state that ServeLive() must be called.

MT + CQ onConfigLoadError produces redundant log lines pkg/statusreconciler/controller.go:204

The onConfigLoadError callback logs "Config load failed; marking status-reconciler unhealthy." with the error. The config agent's own goroutine also logs the same error with "Error loading config.". Every config failure produces two error lines about the same event, which could confuse on-call.

Remove the log line from onConfigLoadError (the health flip is the important action), or change it to Warn to differentiate from the agent's Error.

CQ + MT init() MustRegister can panic in tests pkg/statusreconciler/metrics.go:38-40

prometheus.MustRegister in init() will panic if metrics are already registered (e.g., tests importing the package alongside another package with the same metric name). Follows existing Prow patterns, so consistent and not blocking.

Additional Findings

Medium Integration test fake servers missing ServeLive() Deployment Risk test/integration/cmd/fake*/main.go

Five test fake servers (fakepubsub, fakegitserver, fakeghserver, fakegerritserver, fakegcsserver) call NewHealth() but not ServeLive(). After this change, they won't have a /healthz endpoint at all. If integration tests rely on liveness probes for these, they'll get 404s.

Audit these five files and add ServeLive() calls where needed.

Medium ConfigAgentWithAdditionals removal is a breaking API change Deployment Risk pkg/flagutil/config/config.go

ConfigAgentWithAdditionals and the old ConfigAgent(reuse ...*config.Agent) signature are both removed. Downstream Go consumers importing these will fail to compile. The breaks are compile-time (not silent runtime), but should be documented.

Document in release notes. The functional options pattern is the replacement.

Low Health recovery semantics are implicit Maintainability + Code Quality pkg/statusreconciler/controller.go:179

c.setHealthy(true) on every channel receive works because a Delta only arrives after successful config load (Agent.Set() publishes the delta). This depends on understanding config.Agent internals. No log message on recovery makes debugging liveness flaps harder.

Add a comment explaining recovery semantics. Consider adding a log line when transitioning from unhealthy to healthy.

Low Run exits permanently on Load failure — no retry Maintainability pkg/statusreconciler/controller.go:167-172

When Load() fails, Run logs and returns. The goroutine silently exits. If the saved status file is transiently unreadable (NFS hiccup, GCS error), the reconciler is permanently dead until the liveness probe restarts the pod. Correct behavior, but the intentional coupling should be documented.

Add a comment: "Run exits on Load failure; the liveness probe will restart the pod."

Low onConfigLoadError: dead nil check on err Code Quality pkg/statusreconciler/controller.go:204-209

onConfigLoadError checks if err != nil before logging, but it's only ever called from paths where err is already non-nil. The unconditional setHealthy(false) means calling with nil would still mark unhealthy.

func (c *Controller) onConfigLoadError(err error) {
    if err != nil {  // always true when called
        logrus.WithError(err).Error("Config load failed; ...")
    }
    c.setHealthy(false)  // runs even if err is nil?
}

Remove the nil check, or guard setHealthy(false) behind it too for consistency.

Low StartWatch now propagates watchConfigs error (undocumented behavior change) Code Quality + Deployment Risk pkg/config/agent.go:316

Previously StartWatch silently discarded watchConfigs return value. Now it returns it. If fsnotify setup fails (e.g., inotify limits), callers will see an error they previously didn't.

Call out this behavioral change in the PR description.

Low StartWithErrorHandler has a 7-parameter signature Maintainability pkg/config/agent.go:351

Adding onError to an already-long parameter list. Pre-existing debt slightly worsened. The functional options pattern used at the ConfigOptions level could eventually replace this too.

Deployment Notes

API Breaking Change: ConfigAgentWithAdditionals removed and ConfigAgent signature changed. Downstream Go consumers must update to functional options (WithReuseAgent, WithAdditionals).
Health API Change: /healthz no longer auto-registered by NewHealthOnPort. Out-of-tree consumers of pkg/pjutil/health.go must add ServeLive().
Behavioral Change: status-reconciler now crash-loops on corrupted state files instead of running with stale/empty config. Intended improvement — operators with a corrupted file will need to fix or remove it.
Behavioral Change: Transient config load failures (NFS glitch, ConfigMap race) now trigger pod restarts via liveness probe instead of running with stale config. This is the desired "fail closed" behavior.
Rollback: Safe. No config file format changes, no persistent state format changes.
