Summary

When trusted_apps (e.g. ["dependabot"]) is configured in the DCO plugin but neither SkipDCOCheckForMembers nor SkipDCOCheckForCollaborators is true, the filterTrustedUsers() call is completely skipped. This makes trusted_apps silently ineffective.

The fix adds len(config.TrustedApps) > 0 to the guard condition so that trusted apps are filtered even without the skip flags.

Side effect: implicit org-member trust

Worth discussing: When only TrustedApps is configured (skip flags are both false), the fix causes filterTrustedUsers to run, which calls trigger.TrustedUser(). That function always checks org membership first (line 261). This means all org members will also have their DCO check skipped, even though SkipDCOCheckForMembers is explicitly false.

Is this a real problem?

  • In practice, probably not. Repos that care about DCO enforcement for members are unlikely to configure only TrustedApps without the skip flags. The typical use case is: "I have skip flags on, and I also want to trust dependabot."
  • In principle, it's a semantic violation. SkipDCOCheckForMembers: false should mean members are not skipped, but entering the trusted-user path makes that setting a no-op for org members.
  • The root cause is in trigger.TrustedUser(), not in this PR. The function bundles too many checks together. Fixing it properly would require refactoring TrustedUser() or passing additional flags — out of scope for this bug fix.

Possible follow-up

If you want to flag this: Consider suggesting a follow-up issue where filterTrustedUsers or trigger.TrustedUser() is refactored to separate the trusted-apps check from the membership checks. That way TrustedApps could be evaluated independently without side effects on the skip-flag semantics.

Draft PR comment

LGTM. Clean, minimal fix for a real bug where `trusted_apps` was silently
ignored unless skip flags were also enabled. The fix is correct, the test
coverage is appropriate, and deployment risk is negligible.

Two minor suggestions for follow-up (neither blocking):

1. Consider extracting the growing guard condition into a helper method on the
   `Dco` struct (e.g. `shouldFilterTrustedUsers()`) to prevent the same class
   of bug recurring if new trust mechanisms are added.

2. A brief inline comment explaining why `TrustedApps` needs its own
   independent check would help prevent a future contributor from
   "simplifying" the condition back.

Note: when only `TrustedApps` is configured (both skip flags false),
`trigger.TrustedUser()` will also implicitly trust org members via its
`IsMember()` check. This is pre-existing behavior in `TrustedUser()`, not
introduced by this PR, but worth being aware of.
