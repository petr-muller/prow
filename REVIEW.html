Required Changes

Critical All 3 reviewers handleStatusEvent ignores org/repo sharding

pkg/plugins/blunderbuss/blunderbuss.go:256

handleStatusEvent passes the full Blunderbuss struct to handleStatus instead of using BlunderbussFor(se.Repo.Owner.Login, se.Repo.Name). Status-triggered reviewer assignment always uses global config, silently ignoring org/repo overrides.

The other two handlers (handlePullRequestEvent at line 167 and handleGenericCommentEvent at line 210) were correctly updated. This one was missed.

Design tension

handleStatus currently accepts plugins.Blunderbuss and accesses config fields (IgnoreDrafts, IgnoreAuthors, ReviewerCount, WaitForStatus, etc.) via the embedded pointer. It iterates over matching PRs from a SHA. The fix isn't a simple parameter swap — the function may need restructuring since it checks WaitForStatus as a precondition.

Fix

  • Change handleStatus to accept BlunderbussConfig instead of Blunderbuss
  • Have handleStatusEvent call BlunderbussFor(se.Repo.Owner.Login, se.Repo.Name)

Critical 2 reviewers Previously flagged Nil pointer dereference in BlunderbussFor

pkg/plugins/config.go:944

The fallback return return *c.Blunderbuss.BlunderbussConfig dereferences without a nil check. While setDefaults() initializes the pointer during normal config loading, BlunderbussFor can be called before that — tests already do this (e.g., blunderbuss_test.go:761 calls BlunderbussFor directly without setDefaults()).

This was flagged by jmguzik in Mar 2025. The author acknowledged it but the fix was never pushed.

Fix

  • Add a nil guard: if c.Blunderbuss.BlunderbussConfig == nil, return a zero-value BlunderbussConfig{}

Concern 2 reviewers validateBlunderbuss skips sharded configs

pkg/plugins/config.go:1268-1273

Validation only checks ReviewerCount at the top level. Org/repo-level sharded configs are not validated. A user could set request_count: -1 in an org override and it would pass silently.

Fix

  • Iterate over Blunderbuss.Orgs and their Repos, applying the same ReviewerCount validation at each level

Design Question: Full Replacement vs Cascading Defaults

Design 2 reviewers BlunderbussFor uses full replacement, not field-level merge

pkg/plugins/config.go:928-945

BlunderbussFor() returns the most specific config wholesale. If global sets request_count: 3 and a repo override only sets ignore_authors: ["bot"], the repo gets request_count: 0 (zero value), not 3.

This differs from the Bugzilla pattern. ResolveBugzillaOptions cascades defaults from parent scopes. The current behavior means operators must duplicate all desired fields at every override level — error-prone for large installations.

The test TestHandlePullRequestShardedConfig demonstrates this: the repo config sets only IgnoreAuthors, so ReviewerCount is nil, causing zero reviewers to be requested. The test passes because it expects 0 — but this is arguably a foot-gun.

Recommendation: This is non-blocking if the behavior is explicitly documented with a code comment. But consider implementing cascading defaults in a follow-up.

BlunderbussOrgConfig embeds *BlunderbussConfig (pointer) while BlunderbussRepoConfig embeds BlunderbussConfig (value). The asymmetry is unexplained. A brief comment on BlunderbussRepoConfig would help future contributors.

Deployment Notes

  • No breaking changes for existing deployments. Flat YAML configs parse identically.
  • Rollback is safe — older binaries silently ignore the orgs key.
  • Operators adopting sharded config: overrides fully replace the parent config, they do not merge field-by-field. Every desired field must be specified at the override level.
  • Status events: until the handleStatusEvent fix lands, per-org/repo overrides are only respected for PR open and /auto-cc events, not status-triggered reviewer assignment.
