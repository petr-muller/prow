PR #706: Add custom KIND_CONFIG support to make dev

github.com/kubernetes-sigs/prow/pull/706 · Author: Daniel-Giszpenc (NeoStudyHat) · +235 / -8 · Request Changes

Adds a KIND_CONFIG environment variable to make dev / make dev-full, allowing users to pass a custom KinD cluster configuration file. The main use case is running the KinD cluster on a remote server and connecting from a dev workstation via Tailscale.

The core plumbing is correctly implemented. @Prucek reviewed on May 5 and flagged 3 issues (all overlapping with our findings). The author pushed a fix commit f282197 addressing those. Prucek approved on May 6 and ran /ok-to-test. Some findings remain open but were deemed non-blocking. Still needs lgtm and final approval from @smg247.

Findings — Converging Concerns (flagged by 2+ reviewers)
Partial print_connection_info degraded for all users CQMT ▶

Update (May 5): The #TODO comment was removed in f282197, but the "Custom KIND_CONFIG Note" block itself is still printed unconditionally for all users. @Prucek flagged this as "probably not supposed to be here anymore" — the author removed the TODO but didn't make the note conditional or remove it. Still needs fixing.

The "APISeverAddress" typo also remains.

Fixed Port options silently ignored with custom config CQMTDR ▶

Resolved in f282197. Author added two warning log lines in dev-env.sh when kind_config is set: log "WARNING: env vars DEV_HTTP_PORT and DEV_HTTPS_PORT are not used when KIND_CONFIG is set." and log "All values are set in the yaml file specified at the KIND_CONFIG path."

Note: the connection info still prints ${DEV_HTTP_PORT} even with custom config, but the warning makes this much less confusing.

Bug No validation that custom config file exists CQ ▶

When a user passes a custom config, it goes directly to kind create cluster --config= at test/integration/setup-kind-cluster.sh:191 without checking that the file exists or is readable. A missing file produces a cryptic kind error instead of a clear message.

One-line fix: if [[ ! -f "${kind_config}" ]]; then log "ERROR: kind config file not found: ${kind_config}"; return 1; fi

Suggestion Custom config silently skips containerdConfigPatches MT ▶

The default heredoc in test/integration/setup-kind-cluster.sh:161-189 includes a containerdConfigPatches block that configures the local registry cert directory. When a custom config is used, this is entirely bypassed with no warning.

Users who forget to include this stanza in their custom YAML will get a broken local registry with no helpful error. Consider emitting a warning: log "WARNING: Using custom KIND config — ensure containerdConfigPatches for /etc/containerd/certs.d is included for the local registry to work"

Fixed Sentinel value inconsistency between scripts MT ▶

Resolved in f282197. Both scripts now use empty string. setup-kind-cluster.sh changed from kind_config=default to kind_config="", and the conditional changed from [ "${kind_config}" == "default" ] to [[ -z "${kind_config}" ]].

Additional Findings
Suggestion Inconsistent --wait=5m only in custom config path CQ ▶

At test/integration/setup-kind-cluster.sh:191, the custom config path uses --wait=5m, but the default heredoc path does not. If the wait is needed to handle clusters that take longer to come up (likely with remote/custom networking), it should be documented why it's only there. If it's generally useful, consider adding it to the default path too.

Doc Two-machine guide needs revision MTDR ▶

site/content/en/docs/local-dev-two-machines.md

Tone and scope: The doc is quite opinionated (Tailscale, Debian, specific Go version, specific firewall tool) for a project-level guide. It reads more like a personal blog post. Consider whether this belongs in the official site or as a linked community guide.

Security: Exposing the Docker daemon on TCP without TLS is a significant security concern. Docker's own docs strongly recommend TLS for remote daemon access. The current warning is too brief for the risk involved. The scp root@... example also encourages SSHing as root.

Staleness risk: Hardcoded Go version 1.22.12 will go stale. Better to point at go.dev/doc/install.

Phrasing: "ways are available and possibly required based on your systems" is unclear.

Nit Usage help formatting inconsistency CQ ▶

test/integration/setup-kind-cluster.sh:57 uses a semicolon (-kind-config='';) while all other option entries in the same block use a colon. Should match existing style.

Nit dev-env.sh missing -kind-config in usage text DR ▶

The usage() function in hack/dev-env.sh:46-96 documents -profile, -no-build, -rebuild, -teardown, and -help but does not mention the new -kind-config option. Users calling hack/dev-env.sh -help won't discover it.

What's good

  • Makefile syntax ($(if $(strip ...),…)) is correct and idiomatic GNU Make
  • Shell argument parsing follows existing project conventions (single-dash flags, ${arg#-flag=})
  • The feature itself is reasonable — custom KinD configs are a legitimate need for remote dev
  • The Makefile change is minimal and doesn't affect the default path when KIND_CONFIG is unset
  • Change is additive and would revert cleanly
  • --wait=5m on the custom config path is a reasonable safety net

Maintainer Advisor — Final Recommendation

Request Changes

All three reviewers agree this is a low-risk, well-scoped change to dev tooling, but two out of three independently request changes for the same core issues: unconditional output that degrades the default UX, and silent parameter dropping. The fixes are small and straightforward.

Required Changes

  1. Gate the "Custom KIND_CONFIG Note" behind a conditional — only print when kind_config is set. Ship it fixed, not with a TODO.
  2. Warn when port env vars are ignored — when KIND_CONFIG is set alongside DEV_HTTP_PORT/DEV_HTTPS_PORT, log a clear warning.
  3. Validate the custom config file exists — one-line check before passing to kind create cluster.
