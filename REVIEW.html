Converging Concerns

Converging — All 3 Reviewers
Scheduling modes: minimum_interval is a 3rd mutually exclusive mode, not a modifier
horologium.md:15-19
Flagged by: Code Quality Maintainability Deployment Risk

The doc presents two scheduling modes (interval and cron) and describes minimum_interval as a modifier on interval. Config validation at pkg/config/config.go:2428-2429 explicitly enforces that cron, interval, and minimum_interval are mutually exclusive. An operator reading this doc might try to set both interval and minimum_interval on the same job, which would be rejected.

Suggested rewrite:

Horologium supports three mutually exclusive scheduling modes for periodic jobs:

* `interval`: run after the configured duration has elapsed since the
  previous run's start time, provided the previous run has completed.
* `minimum_interval`: run after the configured duration has elapsed since
  the previous run's completion time.
* `cron`: run when the configured cron schedule fires, provided the
  previous run has completed.

Retry hard cap of 10 not documented
horologium.md:27
Flagged by: Code Quality Maintainability

The doc says "until the configured attempt limit is reached". The code also applies a hard cap of maxRetries = 10 (main.go:51-52) via min(p.Retry.Attempts, maxRetries). Operators configuring more than 10 retry attempts would be silently capped.

Suggested wording:

If a periodic job config includes retry settings, Horologium can also create
retry ProwJobs until the configured attempt limit is reached (hard-capped at 10).

Draft PR Comment

Good documentation that accurately describes Horologium's behavior — a clear improvement over the placeholder. All defaults and claims verified against source code. One issue worth addressing: the scheduling section presents minimum_interval as a modifier on interval, but config validation enforces these as three mutually exclusive modes (interval, minimum_interval, cron). As written, an operator might try combining interval and minimum_interval on the same job, which would be rejected. I'd suggest restructuring that section to list three separate modes. Minor: the retry description could mention the hard cap of 10 from maxRetries.
