Verdict

Approve — good to merge as-is or with the two suggested improvements below.

Clean, well-tested change that adds a generic fallback from sparse checkout to full clone when the sparse clone fails. The runClone extraction is the right structural move, the retry orchestration in Run() is straightforward, and the tests cover both the fallback and happy paths using real git repos. No critical issues, no deployment risk, no configuration or API surface changes. Two "should-fix" items are worth the author's consideration but neither is blocking.

What this PR does

  • Extracts the inner clone logic from Run() into a new runClone() function that returns a Record.
  • When a clone fails and sparse checkout was enabled, wipes the clone directory and retries with a full clone (no sparse checkout, no blobless filter).
  • Merges the command history from both the failed sparse attempt and the successful full clone into a single Record for debugging visibility.
  • Adds two integration tests: one that forces a sparse checkout failure via filesystem sabotage and verifies the fallback succeeds, and one that confirms the happy path still works.

Findings

Should fix

record.Refs reflects fallback refs, not original request
pkg/pod-utils/clone/clone.go:75-88

After fallback, record = runClone(fullRefs, ...) overwrites record.Refs with fullRefs, which has SparseCheckoutFiles: nil. Any downstream consumer inspecting Record.Refs.SparseCheckoutFiles will not know sparse checkout was originally requested. The PR description explicitly calls out debugging visibility as a goal, so the record should reflect the original request.

Fix: add record.Refs = refs after the fallback runClone call.

record = runClone(fullRefs, dir, gitUserName, gitUserEmail, cookiePath, env, user, token)
record.Commands = append(failedCommands, record.Commands...)
// add: record.Refs = refs
os.RemoveAll failure proceeds into doomed retry
pkg/pod-utils/clone/clone.go:81-83

If os.RemoveAll(cloneDir) fails (e.g. permission denied), the code logs a warning but proceeds to runClone into a dirty directory. The second attempt will almost certainly fail with a confusing, unrelated error that obscures the real problem. Consider returning the original failed record when cleanup fails. 

if err := os.RemoveAll(cloneDir); err != nil {
    logrus.WithError(err).Warn("Failed to clean up clone directory for fallback")
}

Nits

Missing comment explaining why fallback exists
pkg/pod-utils/clone/clone.go:77

The code is clear about what it does but not why. A one-line comment noting the submodule/gitlink edge case would save future maintainers from needing to find this PR.

Fallback test could assert record.Failed more directly
pkg/pod-utils/clone/clone_test.go:999

The test checks record.Failed via an if branch but doesn't have a standalone assertion that makes the success contract immediately scannable. Minor readability improvement.

Things I checked and was fine with

  • runClone extraction is clean; startTime renamed to cmdStart to avoid shadowing
  • fullRefs := refs is a safe struct copy; nil-ing the slice correctly flips isSparseCheckoutSet
  • append(failedCommands, record.Commands...) is safe — separate slice reference
  • Fallback only triggers when sparse checkout is set AND clone failed — no impact on normal clones
  • Tests use real git repos and assert on behavioral outcomes, not brittle implementation details
  • No configuration, API, CLI flag, or ProwJob spec changes — purely internal
  • No breaking changes, safe to roll back, zero upgrade friction for operators
  • Deployment risk is LOW: only observable change is previously-failing sparse jobs now succeed

Open questions for the author

  • Should record.Refs after a successful fallback reflect the original request (with SparseCheckoutFiles) or what was actually executed (without)? The PR description emphasizes debugging visibility, which argues for the original request.
  • Is warn-and-continue the right behavior when os.RemoveAll fails, or should the fallback be skipped entirely in that case?
