1. No config validation for policy values commented on PR converging

There is no validation that github_merge_blocks_policy values are one of ignore, permit, or block. A typo like "blocker" silently falls through the switch statements (no matching case, no default), effectively behaving as "ignore" (no case matches, nothing happens, merge proceeds with no warning).

This has security implications: an operator who configures "block" with a typo will believe merges are blocked when they are actually permitted.

PR comments posted:

  • tide.go:256 — Config load and checkconfig should validate the value is one of the three.
  • github.go:647 — Do something loud in default: for unexpected values that somehow slip through (programmer error or botched validation).

4. Warning log fires every sync cycle per BLOCKED PR concern

Under the default "permit" policy, poolStatus() emits a warning log for every BLOCKED PR on every status sync cycle — not just once or on state transitions. For installations with many repos using branch protection/rulesets where PRs commonly have BLOCKED status, this could produce significant log noise.

Likely overstated: Under "permit", the PR is still allowed to merge, so it won't linger in the pool. The noisy logging window is just however many sync cycles pass between entering the pool and Tide merging the PR. The real noise scenario would require many PRs in the pool simultaneously with BLOCKED status, which is unlikely in practice.

Original suggestion: Consider a Prometheus counter metric (tide_merge_blocked_prs{org, repo, policy}) instead of or in addition to the warning log.

pkg/tide/status.go:389-405

5. Dual enforcement points lack cross-references suggestion

The BLOCKED state check with policy resolution appears in both isAllowedToMerge() (merge gating) and requirementDiff() (status reporting). These serve different purposes and are functionally correct — with "block" policy, isAllowedToMerge filters the PR out early so requirementDiff is never reached.

However, a future maintainer trying to understand where blocking actually happens could be confused. A brief comment in each location pointing to the other would help. Alternatively, consider extracting a shared helper if the policy resolution logic grows.

6. BLOCKED diff weight same as milestone mismatch (100) question

In requirementDiff, the BLOCKED state adds diff += 100, which is the same weight as a milestone mismatch. Since BLOCKED represents a hard GitHub-side enforcement (branch protection, rulesets), should it have a higher weight?

Currently, if both milestone and BLOCKED are wrong, the milestone description wins because it's checked first and desc is already set. The user would see "Must be in milestone X" instead of the more actionable "Blocked by GitHub" message.

11. Config comment about "ignore" appears backwards commented on PR concern

The config documentation says: "Note: 'ignore' may cause issues with repos using GitHub Rulesets that restrict updates but have Tide on the bypass list."

This appears backwards. Per issue #575, the problem is with "block" policy: when a repo uses a "restrict updates" ruleset and Tide is on the bypass list, GitHub still reports mergeStateStatus as BLOCKED. With "block" policy, Tide refuses to merge even though it can (bypass list). With "ignore", Tide would correctly proceed. The comment warns about the wrong policy.
