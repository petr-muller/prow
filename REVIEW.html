Issues

Issue

1. locked: true at repo level is silently accepted but meaningless

pkg/plugins/config.go — resolveTrustedApps(), validateTrustedApps()

resolveTrustedApps correctly doesn't propagate locked from repo-level entries (there is no lower level to lock against), but validateTrustedApps doesn't warn about it. A user setting locked: true on a repo-level trigger gets no feedback that it has no effect.

Consider adding a validation warning, or at minimum documenting this behavior.

Issue

2. Multiple matching triggers at same level: subtle app merging vs. first-match for other fields

pkg/plugins/config.go — TriggerFor()

If two trigger entries both match the same org, their TrustedApps are merged (via append), but only the first trigger's other fields (OnlyOrgMembers, TrustedOrg, etc.) are used. This is likely intentional and matches the pre-existing behavior for non-trusted-apps fields, but it's a subtle contract worth a code comment.

Things done well

Good

Backward-compatible unmarshaling

Custom UnmarshalJSON handles legacy strings, new objects, and mixed arrays elegantly. Since sigs.k8s.io/yaml delegates to JSON internally, no separate YAML unmarshaler is needed — and this is verified by TestTrustedAppsUnmarshalYAML.

Good

GetTrustedApps() accessor pattern

The unexported resolvedTrustedApps field with a public getter forces all callers through TriggerFor resolution. This makes it hard to accidentally use unresolved apps.

Good

Optional checkconfig warning (not error)

ValidateLockedTrustedApps as an optional warning is the right severity — locked entries simply take precedence, so the config isn't broken, just potentially confusing.

Good

Comprehensive test coverage

Tests cover all resolution scenarios: global-only, org overrides, repo overrides, locked preventing overrides at both levels, three-level merging, non-matching orgs, empty config, legacy/new/mixed unmarshaling formats, validation of duplicates and empty names.
