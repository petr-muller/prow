Verdict

REQUEST CHANGES — The new MergeCommitBlocker config must be registered in HasConfigFor() before merge.

The PR is well-structured, follows established codebase patterns, and is fully backwards-compatible for unconfigured installations. Code quality is solid. The one blocking issue is that the new config type is not wired into the configuration sharding mechanism that all sibling configs participate in, which would cause silent misconfiguration in multi-shard Prow deployments.

What this PR does

  • Adds a MergeCommitBlocker config struct with Repos and ExcludedPaths fields, following the same shape as Lgtm, Trigger, etc.
  • Adds two new methods to the git Interactor interface: MergeCommitSHAsBetween (enumerate merge commit SHAs) and CommitChangedFiles (list files changed by a commit).
  • Extends the mergecommitblocker plugin to check whether merge commits touch only excluded paths — if so, they are allowed through without the do-not-merge/contains-merge-commits label.
  • When no config is present, behavior is identical to before (short-circuit to existing MergeCommitsExistBetween).
  • Adds integration tests using localgit with real merge commits, plus unit tests for the path-matching logic.

Findings

[Blocking] MergeCommitBlocker missing from HasConfigFor()

pkg/plugins/config.go:2371-2460

HasConfigFor() explicitly enumerates every plugin config that uses the Repos pattern: Approve, Lgtm, Triggers, Welcome. The new MergeCommitBlocker follows the same shape but is not registered. In multi-shard Prow deployments, the sharding logic uses this method to determine which shard owns config for which repos. Without registration, the excluded paths config could silently not apply.

Additionally, MergeCommitBlocker must be added to the reflect.DeepEqual check at line 2372 (which determines whether non-repo-scoped global config exists). Follow the Lgtm pattern at lines 2422–2430.

[Should fix] diff-tree -m can produce duplicate filenames

pkg/git/v2/interactor.go:622-634

out, err := i.executor.Run("diff-tree", "-m", "--no-commit-id", "--name-only", "-r", sha)

The -m flag diffs a merge commit against each parent separately, so a file changed relative to both parents appears twice in the output. This is harmless for the current caller (isMergeCommitAllowed just checks regex matches, so duplicates are redundant but correct). However, the method doc says "returns the list of files changed" without mentioning this. A one-line comment explaining the -m behavior would prevent future callers from being surprised.

[Should fix] Missing error test case for MergeCommitSHAsBetween

pkg/git/v2/interactor_test.go:2071-2135

The test table covers "no merges", "one merge", and "multiple merges" but has no error case (where the executor returns an error). The sibling TestInteractor_CommitChangedFiles does include an error case. Should be consistent.

[Nit] Consider Debug-level logging of rejected file list

pkg/plugins/mergecommitblocker/mergecommitblocker.go:170-172

log.Infof("Merge commit %s is not allowed because it modifies files outside excluded paths", sha)

When a merge commit is disallowed, only the SHA is logged. Logging the actual file list at Debug level would make it easier for operators to diagnose why a PR was flagged.

[Question] Interactor interface expansion — release notes?

pkg/git/v2/interactor.go:77-80

Two new methods are added to the public Interactor interface. Any external code implementing this interface directly (rather than embedding the concrete type) will fail to compile. Is this expected to be documented in release notes?

Things I checked and was fine with

  • Backward compatibility: unconfigured repos hit the fast path (cfg == nil || len(cfg.CompiledExcludedPaths) == 0) and behave identically to before.
  • Regex compilation happens at config load time in compileRegexpsAndDurations with clear error messages; invalid patterns fail fast.
  • CompiledExcludedPaths is tagged json:"-", preventing serialization issues.
  • MergeCommitBlockerFor uses index-based range (&c.MergeCommitBlocker[i]) returning a pointer to the actual slice element, not a loop-variable copy.
  • isMergeCommitAllowed is a pure function, independently testable, with good edge case coverage (empty files, empty patterns, partial matches).
  • Integration tests use real git operations via localgit rather than mocks.
  • Error wrapping is consistent throughout with %w.
  • Config lookup pattern (MergeCommitBlockerFor) matches the established LgtmFor / TriggerFor precedent.
  • No security concerns: SHA values passed to git commands originate from git log output.
  • Deployment risk is low: no config migration needed, safe rollback, negligible performance impact.

Open questions for the author

  • The -m flag on diff-tree diffs against each parent of a merge commit, which means files changed on master between the branch point and the merge appear as "changed." For standard subtree merges this is fine, but have you considered edge cases with unusual merge topologies? Would --first-parent be more precise for this use case?
  • Should the Interactor interface expansion be called out in release notes for external consumers?
