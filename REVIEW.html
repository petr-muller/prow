Approve with suggestions

The fix is correct: it adds ValidateStorageBucket at exactly the right point in ResolveSymlink — after alias resolution, before opener.Reader(). It uses the same validation function already enforced in StorageArtifactFetcher, and is gated by the existing SkipStoragePathValidation opt-in flag, so default installations see zero behavioral change. Two follow-ups are needed (test coverage and a minor accessor inconsistency) but neither blocks merge of a security fix with no deployment risk.

What this PR does

  • Closes a confused-deputy / SSRF in Deck's Spyglass: ResolveSymlink was the only code path that called opener.Reader() on a user-controlled storage key without first checking the bucket against the allowlist.
  • Inserts a 4-line guard that extracts the bucket from the key (post-alias-resolution) and calls sg.cfg().ValidateStorageBucket(bucket); returns a descriptive error on mismatch.
  • Only affects deployments that have opted into storage path validation (SkipStoragePathValidation: false). Default behaviour is unchanged.
  • Originally reported as kubernetes/test-infra#27333; the earlier fix (test-infra#27249) missed this code path.

Findings

Should fix No test exercises the new validation
pkg/spyglass/spyglass_test.go:1504-1513

TestResolveSymlink builds its config without setting SkipStoragePathValidation or AllKnownStorageBuckets. Since SkipStoragePathValidation defaults to nil, shouldValidateStorageBuckets() returns false and the new check is a no-op in every existing test case. The security fix has zero regression protection. Add at least: (1) a case with validation enabled + disallowed bucket → expect error, (2) a case with validation enabled + allowed bucket → expect success.

Nit sg.cfg() vs sg.config() inconsistency
pkg/spyglass/spyglass.go:195

The new line uses sg.cfg(), which is promoted from the embedded *StorageArtifactFetcher (its own cfg field, line 45 of the fetcher). Every other method on Spyglass in this file uses sg.config() (the direct field at line 61). Both are initialised from the same cfg parameter in New(), so they return the same value — but a future reader of spyglass.go would reasonably wonder whether they differ. Change to sg.config().

Question Rejected requests surface as HTTP 500, not 400
cmd/deck/main.go (renderSpyglass → httpStatusForError)

The error from ResolveSymlink is wrapped by renderSpyglass in a plain fmt.Errorf. httpStatusForError looks for an httpError wrapper and falls back to 500 when it doesn't find one. The existing validateStoragePath in the same file wraps bucket-validation errors in httpError{statusCode: http.StatusBadRequest}, so those return 400. Should the author address this here, or is it acceptable as a follow-up?

Things I checked and was fine with

  • Placement: validation is after BucketAliases resolution (line 191-192) and before opener.Reader() (line 198). No I/O for disallowed buckets.
  • Bucket extraction via strings.Split(key, "/")[0] mirrors the alias extraction on line 190 and is correct for the bucket/path... key format here.
  • Double alias resolution (explicit on line 192, then inside ValidateStorageBucket at config.go:1334) is idempotent and harmless — matches the pattern in StorageArtifactFetcher.parseStorageURL.
  • Error wrapping uses %w, message is actionable and includes the bucket name.
  • No deployment risk for default installations: validation is gated by SkipStoragePathValidation, which defaults to nil (validation off).
  • ResolveSymlink was the only unguarded path to opener.Reader() in the spyglass package; StorageArtifactFetcher already validates at line 82.
  • Change is safely rollbackable, no config migration needed.

Open questions for the author

  • Could you add a test case to TestResolveSymlink that enables bucket validation (set SkipStoragePathValidation to ptr(false), populate AllKnownStorageBuckets) and verifies a disallowed bucket is rejected? The existing tests run with validation off, so the new guard is never exercised.
  • Line 195 uses sg.cfg() (from the embedded fetcher) while the rest of this file uses sg.config(). Both return the same config, but for consistency could you switch to sg.config()?
  • The error surfaces as HTTP 500 because renderSpyglass doesn't wrap it as an httpError. The parallel check in validateStoragePath returns 400. Is this something you'd want to address here, or as a follow-up?
