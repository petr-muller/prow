1. No config validation for policy values 3/3 reviewers converging

There is no validation that github_merge_blocks_policy values are one of ignore, permit, or block. A typo like "blocker" silently falls through the switch statements (no matching case, no default), effectively behaving as "permit" with no warning.

This has security implications: an operator who configures "block" with a typo will believe merges are blocked when they are actually permitted. Unlike PrioritizeExistingBatchesMap (which uses bool and gets validation at unmarshal time), this is a free-form string type.

Suggested fix: Add validation at config load time, or at minimum add a default case with an error log in the switch statements:

for key, policy := range t.GitHubMergeBlocksPolicyMap {
    switch policy {
    case GitHubMergeBlocksIgnore, GitHubMergeBlocksPermit, GitHubMergeBlocksBlock:
    default:
        return fmt.Errorf("invalid github_merge_blocks_policy %q for %q", policy, key)
    }
}

4. Warning log fires every sync cycle per BLOCKED PR concern

Under the default "permit" policy, poolStatus() emits a warning log for every BLOCKED PR on every status sync cycle — not just once or on state transitions. For installations with many repos using branch protection/rulesets where PRs commonly have BLOCKED status, this could produce significant log noise.

Suggestion: Consider a Prometheus counter metric (tide_merge_blocked_prs{org, repo, policy}) instead of or in addition to the warning log. A metric is more operationally useful for dashboards and alerting, especially for operators transitioning from "permit" to "block".

5. Dual enforcement points lack cross-references suggestion

The BLOCKED state check with policy resolution appears in both isAllowedToMerge() (merge gating) and requirementDiff() (status reporting). These serve different purposes and are functionally correct — with "block" policy, isAllowedToMerge filters the PR out early so requirementDiff is never reached.

However, a future maintainer trying to understand where blocking actually happens could be confused. A brief comment in each location pointing to the other would help. Alternatively, consider extracting a shared helper if the policy resolution logic grows.

pkg/tide/github.go:638, pkg/tide/status.go:266

Flagged by: Maintainability, Deployment Risk

6. BLOCKED diff weight same as milestone mismatch (100) question

In requirementDiff, the BLOCKED state adds diff += 100, which is the same weight as a milestone mismatch. Since BLOCKED represents a hard GitHub-side enforcement (branch protection, rulesets), should it have a higher weight?

Currently, if both milestone and BLOCKED are wrong, the milestone description wins because it's checked first and desc is already set. The user would see "Must be in milestone X" instead of the more actionable "Blocked by GitHub" message.

7. requirementDiff now takes full *config.Tide — increased coupling suggestion

The requirementDiff function gained a *config.Tide parameter. The nil-check at line 266 (if tide != nil) suggests it can be nil, but the caller always passes &sc.config().Tide which is never nil. The nil path exists only for test convenience.

Passing the full config increases coupling. An alternative would be to pass just the resolved GitHubMergeBlocksPolicy for the relevant repo, keeping the function's dependency narrow. Acceptable for now but worth noting if more config-dependent checks are added.
