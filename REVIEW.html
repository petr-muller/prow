Verdict

Approve with suggestions

A clean, well-scoped fix for a real user-facing hang. All three review perspectives (code quality, maintainability, deployment risk) converge on low risk and correct implementation. Two converging concerns (Callback path inconsistency, missing CSS) are worth addressing but not blocking. The error-surfacing improvement is a net positive for both users and operators.

Refresh (2026-05-14): Author force-pushed addressing review feedback. Two [should-fix] findings resolved (Size() error handling, missing CSS). Added a UI warning banner when highlighting is skipped. One [should-fix] (error message stutter) and three [nit]s remain open.

What this PR does

  • Adds a 10 MiB size threshold: artifacts larger than this skip regex-based syntax highlighting in Body(), preventing the browser from hanging on pathological regex matching. Lines still render, just without colored highlights.
  • Surfaces ReadAll() failures in the UI: previously, a failed artifact was silently dropped via continue; now it appends a LogArtifactView with an error message so the user sees what went wrong and which artifact failed.
  • Adds a nil-regex guard in highlightLines() to prevent a nil-pointer dereference when highlightRegex is nil, making the function safe for any caller.
  • Adds two new tests covering both behaviors (TestBodyReadAllError, TestBodyLargeLogSkipsHighlight) with an idiomatic errArtifact test helper.

Since previous review:

  • Size() error path now disables highlighting conservatively and logs a logrus.Warn.
  • Added .log-error and .log-warning CSS rules in buildlog.css.
  • Added a Warning field to LogArtifactView and renders a yellow warning banner when highlighting is skipped ("log exceeds 10 MiB" or "unable to determine log size").
  • New test TestBodySizeErrorDisablesHighlight and extended assertions in TestBodyLargeLogSkipsHighlight.

Resolved

[Should fix] Missing CSS for .log-error

buildlog.css:139-147

Added .log-error (red) and .log-warning (yellow) CSS rules.

Open

[Should fix] Error message stutter

lens.go:230 combined with lens.go:493

// lens.go:230
av.Error = fmt.Sprintf("Failed to read log: %v", err)

// lens.go:493 (logLinesAll)
return nil, fmt.Errorf("failed to read log %q: %w", artifact.JobPath(), err)

The user sees "Failed to read log: failed to read log "build-log.txt": file size over specified limit" — "failed to read log" appears twice. Either change the outer message to just "Error: %v", or unwrap the inner message.

[Nit] Callback path does not apply size guard

lens.go:452, 469

logLines := highlightLines(skipLines, skipRequest.StartLine, &request.Artifact, conf.highlightRegex, conf.highlightLengthMax)
// ...
logLines := highlightLines(lines, request.StartLine, &request.Artifact, conf.highlightRegex, conf.highlightLengthMax)

The loadLines function (Callback path for expanding collapsed line groups) always passes conf.highlightRegex directly. For a large artifact, expanding hidden lines would still trigger regex matching. Practical risk is lower since callbacks operate on chunks, but it creates an inconsistency: initial render has no highlights, but expanding a section does. Fine as a follow-up.

[Nit] Test could directly verify highlighting is skipped

lens_test.go:1186

The test now asserts warning presence (good), but still doesn't check that "match-highlighted" is absent from the output. Asserting no highlighted spans would directly test the intended behavior.

[Nit] No log line when size threshold triggers

lens.go:244-246

The Size() error path now logs a warning (resolved), but the normal sz > highlightSizeThreshold path still has no server-side log. A logrus.Info with artifact name and size would help operators.

Things I checked and was fine with

  • XSS safety: html/template is used, so {{.Error}} and {{.Warning}} are auto-escaped. No injection risk.
  • Zero-value backwards compatibility: Error and Warning default to "", so {{if .Error}} / {{if .Warning}} are naturally false for all existing successful artifacts.
  • No config/API/deployment impact: LogArtifactView is an internal template struct, not serialized or user-facing config. No schema changes.
  • Test coverage: All new code paths tested. errArtifact helper extended with sizeErr field.
  • Constant placement: highlightSizeThreshold is grouped with the other lens constants, well-named, and commented with its unit.
  • Existing tests pass: All tests in the package pass, including the existing TestBody table-driven tests.

Open questions for the author

  • The Callback path (loadLines) still uses conf.highlightRegex directly — is that intentional for this PR, with a follow-up planned?
  • The warning message hardcodes "10 MiB" — would it be better to derive from the constant?
