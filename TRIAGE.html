Verdict

Accepted. The issue is partially implemented. PR #612 added workflow approval on /ok-to-test comment only. Workflow approval does not persist when new commits are pushed to trusted fork PRs. The issue should be reopened.

PR #612 merged a one-shot approval mechanism that fires exclusively on the /ok-to-test comment. The trigger plugin's PR event handlers (synchronize, label-added, reopen) start Prow presubmit jobs for trusted PRs but never call approveGitHubActionsWorkflowRuns(). This leaves GitHub Actions workflow runs unapproved when new commits are pushed to fork PRs that already carry the ok-to-test label. Two maintainers (@sbueringer, @stevehipwell) confirmed the gap on 2026-05-13/15 and asked to reopen.

What the issue reports

  • Fork PRs from new contributors require manual approval for GitHub Actions workflow runs, even after a maintainer marks the PR as trusted via /ok-to-test.
  • The original request (from kubernetes/test-infra#25210) asked Prow to call GitHub's workflow run approval API when ok-to-test is applied.
  • PR #612 implemented approval on the /ok-to-test comment, but the approval does not persist across new commits pushed to the PR.
  • Multiple communities (Kubernetes, Volcano, Kubeflow) rely on workarounds (separate GitHub Actions to auto-approve) because the built-in support is incomplete.
  • @sbueringer confirmed on 2026-05-13 that he still had to approve actions manually after new commits were pushed on ok-to-test'd PRs.

Analysis

Probable cause

approveGitHubActionsWorkflowRuns() is called from exactly one place: the /ok-to-test comment handler in generic-comment.go:146-154. The PR event handlers that fire on new commits (PullRequestActionSynchronize) and label additions (PullRequestActionLabeled) call buildAllIfTrusted() or buildAllButDrafts() to start Prow jobs, but never invoke workflow approval.

When a new commit is pushed to a fork PR with the ok-to-test label, GitHub creates new workflow runs in action_required status. Prow starts its own presubmit jobs (trust check passes), but the GitHub Actions runs remain blocked.

Related code

The only call site — pkg/plugins/trigger/generic-comment.go:146-154:

if isOkToTest && trigger.TriggerGitHubWorkflows {
    headSHA, err := refGetter.HeadSHA()
    if err != nil {
        c.Logger.Warnf("headSHA unavailable, cannot approve pending workflows: %v", err)
    } else {
        approveGitHubActionsWorkflowRuns(c, org, repo, pr.Head.Ref, headSHA)
    }
}

Natural injection point — pkg/plugins/trigger/pull-request.go:227-249, buildAllIfTrusted(): already receives the trigger config (to check TriggerGitHubWorkflows) and determines trust. Adding a call to approveGitHubActionsWorkflowRuns() here covers synchronize, reopen, edit, and ready-for-review events.

Approval helper — pkg/plugins/trigger/generic-comment.go:301-337: lists pending runs via GetPendingApprovalActionRuns() and approves each in a goroutine. Handles 404 (already approved) and 403 (permission denied) gracefully.

Config flag — pkg/plugins/config.go:545-546: TriggerGitHubWorkflows bool (trigger_github_workflows in JSON). Feature is opt-in.
